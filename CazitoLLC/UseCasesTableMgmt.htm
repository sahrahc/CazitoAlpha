New User - Join Table
Use Case Number Use Case Name Status Descripton Details
1.1 New user first to join table Verified New user joins a casino table that does not have any players. User cannot start a game until at least one additional user joins.  1) Start game option is disabled
2) Leave table option is enabled
3) Message 'waiting for another user' displayed to user
4) Game play buttons fold, check, raise, call are positioned around seat but disabled.
1.2 New user joins inactive table Verified New user joins a casino table that has one or more users waiting for play to start, there is no active game, and not all seats are taken. The user is given the next available seat by position. The waiting user is notified that another user joined. Any of the users may start the game. 1) User is given the next available seat
2) Start game button is enabled
3) Leave table is enabled
4) Message that game can be started displayed
5) Game play buttons fold, check, raise, call are positioned around seat but disabled.
6) Game boards for all other players in the table are updated with joined user.
7) Start game button enabled for other players in table.
8) Message that game can be started displayed to all other players in table.
1.3 New user joins inactive but full table Verified New user joins a casino table where every seat is taken but there is no one in the waiting list. Users are waiting for game play to start. New user is placed first on the waiting list. 1) Game boards for user updated with seated players info
2) User notified he/she on waiting list
3) User notified one player on waiting list to user
4) Message with number of players in waiting list updated for all other players in table
5) TODO: show location on waiting list
1.4 New user joins inactive table with waiting list Verified Same as previous case but user placed at end of waiting list. Same as previous case
1.5 Game started on table with waiting users (see game play use cases for other game start scenarios) Verified One of the seated users starts game. Game boards for all players, seated and waiting, are refreshed on game start and after every move or timeout action. 1) See game play use case for details on seated users
2) Waiting users can see whose turn it is, move type, move amount and player status
3) Waiting users can see pot size and community cards
1.6 New user joins active table without waiting list Verified New user joins a table that has empty seats but there is an active game being played. The new user is given a seat; he can observe the game play but is not part of the game until a new game starts. 1) User is given the next available seat
2) Start game button is enabled
3) Leave table is enabled
4) Game boards for all other players in table updated with new waiting user info
1.7 New user joins active table with waiting list Verified New user joins a full table with active game play and a waiting list. The new user's game board is updated with the current game play and refreshed with each move. All other players in table are notified of the new waiting list size. 1) User notified he or she in waiting list
2) User notified of waiting list length
3) User's game board updated with game current status
4) User's game board updated with every game play
5) All other table players (seated and waiting) updated to new waiting list size.
3) TODO: show user's place in waiting list
1.8 New user joins table after game ends Verified Same as 1.5 or 1.6 but game status includes hand info for all players. Same as 1.5 or 1.6 but game status includes hand info for every player, winner and updated stakes for all players.
1.9 New user closes browser while playing a game and rejoins the table Verified New user closes browser while playing an active game. He rejoins the table before his turn time outs and rejoins game play as an active player. 1) User's game board refreshed with current game status upon re-entry
2) User's game board shows next player's turn whether his or another player's
3) Other player's game boards not updated on exit or re-entry
1.10 New user on waiting list closes browser and rejoins a game. Verified New user closes browser while on waiting list. He rejoins the table before a table becomes available to him/her and resumes his place on the waiting list. He is not placed at the end of the queue. 1) User's game board refreshed with current game status upon re-entry
2) User's game board shows next player's turn
3) Other player's game boards not updated on exit or re-entry

Leave Table
3.1 Seated player leaves inactive table without waiting list Verified A seated player leaves the casino table before game starts. Game boards for all other players in table are refreshed.  1) Play buttons (call, fold, check, raise) disappear
2) Seat updated as empty seat
3.2 Seated player leaves inactive table with waiting list Verified A seated player leaves the casino table before game starts. First player on waiting list is automatically given a seat. Game boards for all other players in table are refreshed.  TODO: make notification that user got a seat more prominent.
3.3 Seated player leaves active table Verified A seated player leaves a table in the middle of the game. User is taken back to safe saloon. Game boards for all remaining players are updated to show that the player left and he no longer has a turn in the game (see game play rules). A seat offer is made to the first player in waiting list.  
3.4 Player who is a dealer leaves active table Partly Verified A dealer leaves a table in the middle of the game. All rules in 3.2 apply. Card dealing unchanged; source of cars is same seat. BUG: fix source of dealing not to change to the next player.
3.5 Waiting user leaves active or inactive table  Verified A waiting user leaves an inactive table and is taken back to the salon he came from. Remaining players at table notified of updated size of waiting list. Waiting players after user moved up one place.  
3.6 New user joins inactive table with decreased waiting list Verified Regression - same as 1.2  
3.7 New user joins active table with decreased waiting list Verified Regression - same as 1.4  
3.8 New game started after game end on table with replaced users Verified Users who joined during the previous game are included in the new game as active players. 1) Play buttons for all waiting users show up as disabled (unless turn) on all player's game boards.
3.9 New game started after game end on table with decreased waiting list Verified Game starts with same seated players.  
3.10 In-progress game restarted on table with decreased waiting list Verified Regression - same as regular game play  
3.11 In-progress game restarted on table with replaced users Verified Same as 3.8  
3.12 Seated user attempts leaving active table with two seated players and without waiting list  Verified If only two players remain in table and are playing a game, and one of them leaves, the remaining player wins.  
3.13 Seated user attempts leaving active table with two active seated players and at least one waiting but seated player. Verified If only two active players remain in table and one of them leaves, the remaining player wins.  
Take Seat
5.1 Player accepts seat offer Verified A player is offered a seat and given 10 seconds to accept. Upon acceptance, the seat information is updated on all players in the table (seated or in waiting list) and status for user changed to waiting. TODO: timeout not implemented.
5.2 Game starts before seat offer accepted. Verified A player does not accept a seat offer within the alloted time. The player is placed at the end of the waiting list. The next player in the waiting list is offered the seat and given 10 minutes to accept. All other players in table (except second user) not aware of offer/timeout/reoffer.  
5.3 Second seat offer to player times out Not Implemented A player who does not accept or reject a seat within 10 seconds is removed from the waiting list. His is ejected from the table and put in the saloon he came from. All other players in table notified change in waiting list size.  
5.4 User rejects seat offer Not Implemented A player rejects a seat offer. He remains first in the waiting list for as long as he keeps rejecting a seat instead of timing out. The next player in the waiting list is offered the seat upon rejection.  All other players in table (except second user) not aware of offer/rejection/reoffer.  
5.5 Seat offer times out Not Implemented A player does not accept a seat offer before it expires. The next player in the waiting list is offered the seat upon expiration.   
5.6 First player on waiting list closes browser and a seat becomes available before he rejoins Not Implemented A seat offer for a player who has temporarily closed his browser is still available and the offer made when the user rejoins and if the offer has not expired.  
5.7 Player closes browser and misses seat offer before he rejoins Not Implemented A seat offer for a player who has temporarily closed his browser is given to the next player if he does not rejoin before the offer expires. The player is placed at the end of the queue.
