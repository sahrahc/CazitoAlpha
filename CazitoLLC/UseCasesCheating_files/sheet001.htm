Generic rules for all cheating items
0.1 Availability Implemented The list of cheating options are always available to browse and view descriptions of. 
0.2 Actionability Questions on Requirements A cheating option is only actionable if it can be used. We want the user to freely review their inventory.
There is an action panel where the user can make a choice, but only one can be displayed at the time, so it's not usable for long-running items.
0.3 Contextual Use of Hidden Cards Questions on Requirements There is currently no difference between the sleeve and the groove under the table for now. The user may slide out a hidden card from either place.
Need usability details on making an option available contextually. Should the option to use be next to the hidden card display in addition to the action panel? Could there be multiple ways to use a hidden card that get activated?
0.4 Status Questions on Requirements Some options take effect on the next game. Some effects may not result in an immediate action (see Sally Spotter's) or be long running (see Sake Oil Marker and antidote)
TODO: The UI needs to show the list of all active and pending actions.
Questions:
If the outcome is not as expected by a user, should there be a message? Otherwise the user may think his or her item did not get applied correctly.
0.5 Precedence Questions on Requirements TODO: determine whether multiple but similar options can be done at the same time (e.g., Riverbend Redo and Shelvin's Shuffler's) or whether applying one disables the other. Also need to determine which options are "similar." If anything goes, need to determine the effects in combination, timing may be key.
0.6 API request object Implemented All requests include the event type, user player id and session id.
A card selected from a list is identified in the request by the card label, not by an id. The required bandwidth is negligible and the string comparison is only from among the list of hidden cards for a specific player. Labels are less complex than indexes or id's because any of the hidden cards may be used at any time.
0.7 API response object Implemented Responses are asynchronous and there may be multiple responses, one or none.
The cheaterCardDTO is a special separately from PokerHand or GameCards object which is used to partially (e.g., rank) or completely reveal community or player cards. Community cards have player -1.

1.6 Shelvin's Shuffler Questions on Requirements The shelvin's shuffler is activated once per game but has two parts. The user may view the river card to be dealt and choose to swap to not. The user will not be able to see what the replacement card is going to be until the swap is done.
- The user should apply this item as early as possible as he may be able to place a bet if the user knows what the river card is before anyone else.
- Multiple users may use this item, in which case the timing is key.
- Future: if the user chooses to look first, the river card will show up on the Next Card section. The UI will present a contextual button to swap by clicking on the card.
